# GECAM 每日文献综述站（daily-review）— 设计文档

- 日期：2026-07-18
- 状态：已通过讨论评审，待用户最终确认后进入实现计划（writing-plans）
- 领域：高能天体物理 / 多信使多波段暂现源（对齐 GECAM 科学目标）

---

## 1. 背景与目标

为 GECAM 团队搭建一个**每日自动收集、筛选并综述**新文献的网站。每天在固定时间（北京时间 10:00）自动运行，产出**三级内容**：

1. **单篇综述卡片** —— 每篇入选论文一张卡片，简介内容、团队、亮点。
2. **当日总览综述** —— 对当天全部入选文献做整体综述：概览、亮点、趋势与联系。
3. **累积知识库** —— 论文 / 作者 / 团队三类实体互相链接、随时间越攒越厚。

参考对标：`pubscholar.cn/headlines`（内容偏少）、Cornell FRB newsletter（手动收集）。本项目要用 AI + 自动化做到**更全面的覆盖、尽量不遗漏**。

## 2. 范围与非目标

**范围**
- 领域限定：高能天体物理、多信使多波段暂现源，具体以第 5 节 GECAM 科学画像为准。
- **综述基于全文，不局限于摘要**：入选论文尽量获取并精读全文（arXiv 提供 PDF/HTML/源码，多数相关论文都在 arXiv），仅在全文不可得（纯付费墙期刊且不在 arXiv）时退回摘要。
- 中文输出。
- 内部共享起步（B），架构与部署方式保证可平滑走向公开（C）。

**非目标（本期不做）**
- 不做用户账号 / 权限系统（静态站，只读）。
- 不做期刊官网直接爬虫（列为 Phase 3 / C 阶段）。
- 不追求作者消歧 / 团队识别 100% 准确（见第 12 节已知限制）。

## 3. 用户与部署形态

- **受众**：先课题组内部共享（B），做好后走向公开（C）。
- **仓库**：**公开 GitHub 仓库**（Actions 无限量免费、Pages 免费；代码不含密钥）。
- **站点**：GitHub Pages 静态站，有稳定网址可分享。

## 4. 数据源策略

**当前（Phase 1–2）：arXiv + NASA ADS 双源。** 放弃逐个爬期刊官网（脆弱、易违反 ToS，且 ADS 已聚合）。

| 源 | 角色 | 说明 |
|---|---|---|
| arXiv API | 预印本主入口（最快） | 按分类拉每日新提交，含 cross-list |
| NASA ADS API | 已发表期刊补全 | 聚合 Nature/Science/ApJ/A&A/MNRAS/RAA 等，覆盖"不发 arXiv"的少数论文 |

**全文获取（不局限于摘要）**：对入选论文尽量取全文供精写阶段精读——arXiv 提供 PDF / HTML(ar5iv) / LaTeX 源码，优先用 HTML 或源码（比 PDF 干净好解析）；纯付费墙期刊论文若不在 arXiv 上则退回摘要。全文解析为独立步骤，抽取正文（可按需只保留 引言/方法/结果/结论 主干以控 token）。

**可插拔适配器（关键架构约束）**：数据源实现为统一接口的 source 适配器，主流水线不感知具体来源。**Phase 3（= C 阶段）新增期刊官网爬虫时，只新增一个适配器模块，不改主流程。**

召回优先策略：宽网抓取，宁可混入无关，也不遗漏（见第 6 节打分分层）。

## 5. GECAM 科学画像（相关性判定依据）

固化为配置，供"相关性打分"阶段作为判据。分三层：

**🎯 核心主题（GECAM 直接科学目标）**
- 伽马暴 GRB（长/短暴、prompt/余辉、能谱、jet、宿主）
- 引力波电磁对应体（BNS/NSBH 并合、kilonova、GW170817-like、O4/O5 后随观测）
- 磁星 / 软伽马重复暴 SGR（巨耀发、暴发、SGR 1935+2154）
- 快速射电暴 FRB（尤其高能对应体、FRB–磁星联系）
- 多信使触发与联合（LIGO/Virgo/KAGRA 引力波、IceCube 中微子）

**🔭 相关主题（高能暂现源大类 + 相关任务）**
- 高能暂现天体：TDE、X/γ 射线暂现、新星/超新星激波、中子星/黑洞暂现现象
- 太阳耀斑、硬 X 射线爆发；地球伽马闪 TGF
- 相关任务论文：Fermi/GBM、Swift、HXMT/Insight、Einstein Probe(EP)、SVOM、Integral、Konus-Wind 等，以及 GECAM 自身

**〰️ 边缘主题（宽网兜底，折叠展示，不丢弃）**
- astro-ph.HE 内其余物理（宇宙线、暗物质、AGN 稳态物理等）

**抓取宽网**：arXiv 取 `astro-ph.HE`（含 cross-list）+ `gr-qc` + 选择性 `astro-ph.SR` / `astro-ph.CO`；ADS 按上述主题词 + 相关期刊拉当天已发表。

## 6. 每日流水线

每天北京时间 10:00（02:00 UTC）触发，顺序执行：

```
0 抓取(宽网) → 1 去重归一 → 2 相关性打分 → 2.5 全文获取 → 3 单篇综述 → 4 实体建档 → 5 当日总览 → 6 生成静态站 + 提交
```

- **Stage 0 抓取**：arXiv API + ADS API（Phase 1 仅 arXiv）。抓最近一批新公布论文。
- **Stage 1 去重归一**：合并 arXiv / ADS 记录，按 DOI / arXiv-id / 标题归一去重，形成统一论文记录。
- **Stage 2 相关性打分**（粗筛层）：逐篇（**仅用标题 + 摘要，控成本**）让模型对照第 5 节画像输出 `{相关性分数, 主题标签[], 一句话理由}`。**不硬删**，据分数分核心 / 相关 / 边缘三层。
- **Stage 2.5 全文获取**：对入选论文取全文并抽取正文（arXiv HTML/源码优先，退回 PDF；不可得则退回摘要）。
- **Stage 3 单篇综述**（精写层）：对入选论文（核心 + 相关，边缘可按配置决定是否精写）**基于全文**（不可得时退回摘要）按第 7 节模板生成卡片。
- **Stage 4 实体建档**（精写层）：
  - 抽取**关键作者**（第一 + 通讯 + 资深/末位作者），**新面孔**则拉其全部文章（ADS/arXiv API）生成作者画像；已知作者增量更新。
  - 识别**团队**（PI 锚点 + 具名合作组），生成 / 增量更新团队画像。
  - 写入持久化作者库 / 团队库。
- **Stage 5 当日总览**（综述层）：读入当天全部入选摘要，生成概览 + 亮点 + 趋势（第 7 节模板）。
- **Stage 6 生成 + 提交**：渲染静态页面、更新数据文件、`git commit` → Pages 自动部署。

**幂等 / 容错**：单篇失败不阻断整批；已生成的实体画像可复用避免重复调用；每日运行可重试。

## 7. 内容模板（四类）

### 7.1 单篇综述卡片
- **标题**：英文原题 + 中文译名
- **团队**：第一作者、通讯作者、主要机构；是否知名合作组（LIGO/Fermi 等）
- **标签 + 相关性**：主题标签（GRB/FRB/磁星…）+ 相关性评分 + 所属层（核心/相关/边缘）
- **一句话核心（TL;DR）**
- **内容综述**（3–5 句）：研究对象 / 数据方法 / 主要结果 / 结论
- **亮点**：为什么值得关注、创新点
- **与我们的关联**：与 GECAM 科学目标或团队已有工作的联系（有则写）
- **链接**：arXiv / DOI / ADS
- 卡片内**人名、团队名可点**，跳作者页 / 团队页

### 7.2 当日总览综述（页面顶部）
- **今日概览**：共收录 X 篇（核心 Y / 相关 Z / 边缘 W），各主题分布
- **今日亮点**：AI 精选 2–4 篇，逐条说明为什么亮
- **趋势与联系**：共同趋势、与近期/经典研究的呼应、值得注意的方向
- **动向**（可选）：活跃团队 / 合作组 / 观测设施

### 7.3 作者画像
- 姓名、主要机构、（如有）ORCID
- 研究方向概述（AI 依据其全部文章综述）
- 代表作 / 高被引方向
- 与 GECAM 科学的相关点
- 在本站出现过的论文列表（累积）

### 7.4 团队画像
- 团队名（PI 课题组名 / 具名合作组）、PI、主要机构
- 团队研究方向概述
- 主要成员（随共同署名累积补全）
- 在本站出现过的相关论文（累积）

## 12. 消歧与团队识别的已知限制

- **作者消歧**：优先用 ORCID / ADS 唯一标识拉"某人全部文章"；无标识则"姓名 + 机构"启发式，**承认有少量错配**。

## 19. Phase 1.1 修订：同步模型（sync model）

**动机**：Phase 1 首跑（2026-07-18）暴露：workflow 抓"昨天 UTC"且要求 `published` 严格等于该日，但 arXiv 有 1–2 天公告延迟（外加周末不公告），运行时最新可得的 published 日期通常落后今天约 2 天 → 几乎每天抓到 0 篇。本节修订 §6 每日流水线的抓取/归档模型，取代"抓精确的昨天"。

**核心原则：把"发现"与"归档"分离。** 每次运行是一次 **同步（sync）**，而非"生成某一天"。每篇论文永远归档在它自己的**真实 published 日期**下，与被发现的日期无关。

### 19.1 抓取：滚动窗口 + 分页
- `ArxivSource` 新增 `fetch_recent(end_date, days)`：返回 published 日期落在 `[end_date - days + 1, end_date]` 的全部论文。
- **分页**：按 `submittedDate` 降序，`start=0, page, 2*page, …` 翻页，直到某页出现 published < 窗口起点或无更多条目。取代原来 `max_results=300` 的硬上限（几天就超 300）。
- 配置：`FETCH_WINDOW_DAYS = 7`（覆盖公告延迟 + 周末）、`ARXIV_PAGE_SIZE = 100`。

### 19.2 归档：按真实日期 + 回填合并
- pipeline 改为 `sync(run_date, source, llm, store, …)`：
  1. `papers = source.fetch_recent(run_date, FETCH_WINDOW_DAYS)` → dedupe。
  2. `new = store.unseen_ids(...)` 滤掉已综述过的（保证每篇只花一次 AI 成本）。
  3. **并发**对每篇新论文做 打分 →（core/related）全文 + 单篇综述（见 §19.4）。
  4. 按每篇的**真实 published 日期分组**；对每个受影响的日期：加载已存在的 `data/daily/<真实日期>.json`（无则空）→ 若已有内容，先把**当前版综述**压入 `revisions`（见 §19.3）→ 合并新论文 → **重新生成该日期的当日总览** → 保存。
  5. 全部保存后 `mark_seen` 新论文。
- 首页"最新一天" = 有论文的最新真实日期（`list_days()[0]`），**天然落后日历今天约 1–2 天**（arXiv 真实节奏，非 bug）。首页头部标注"数据截至 arXiv 最新公告日 XX-XX"。

### 19.3 版本留档（三层）
1. **git 全量存档**：每次 sync 提交整个 `data/`，每个历史版本都在 git 里，可完整复原。
2. **JSON 内置 `revisions`**：`DayData` 新增 `revisions: list[dict]`，每条 = `{"synced": 运行日, "n_papers": 当时篇数, "review": 当时的 DailyReview.to_dict()}`。回填导致某天内容变化时，把变化前的综述压入。一天通常只在 arXiv 收全前 1–3 天内被回填几次，之后稳定。采用**轻量版**（存历史综述文字 + 篇数 + 同步日；完整整页快照由 git 兜底）。
3. **站点可看**：日期页底部可折叠"修订历史（N 版）"，逐条展示历史综述 + 当时篇数 + 同步日。

### 19.4 并发
- §6 Stage 3 的逐篇处理（打分 + 全文 + 综述）**逐篇独立**，改为**有界并发**（`ThreadPoolExecutor`，`MAX_CONCURRENCY = 6`），墙钟时间约降至 1/5–1/8。逐篇 try/except 已就位，单篇失败不影响其他。并发上限受 opencode 速率/额度约束，勿过高。

### 19.5 受影响文件
`config.py`（窗口/分页/并发常量）、`models.py`（`DayData.revisions`）、`sources/arxiv_source.py`（`fetch_recent` + 分页）、`store.py`（`load_day` 容缺）、`pipeline.py`（`sync` 编排 + 并发 + 分组回填）、`templates/day.html`（修订历史 + 截至日期）、`scripts/run_daily.py`（调 `sync`）、相应测试。
